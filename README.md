# eCornell AI in Finance Work Sessions (May 2026)

Four interactive work sessions that build an AI-powered portfolio management system from scratch — starting with classical Markowitz optimization and ending with a production-grade system featuring adaptive asset selection, sentiment monitoring, and human escalation protocols.

All materials use [Julia](https://julialang.org/) with [Jupyter notebooks](https://jupyter.org/) and synthetic data generated by [JumpHMM.jl](https://github.com/varnerlab/JumpHMM.jl).

## Prerequisites
| Requirement | Version | Notes |
|:------------|:--------|:------|
| [Julia](https://julialang.org/downloads/) | 1.10+ | Tested on 1.12.5 |
| [IJulia.jl](https://github.com/JuliaLang/IJulia.jl) | any | For running notebooks in Jupyter or VS Code |
| [VS Code](https://code.visualstudio.com/) + [Julia extension](https://www.julia-vscode.org/) | recommended | Or any Jupyter-compatible environment |

## Getting Started
```bash
# 1. Clone the repository
git clone https://github.com/varnerlab/eCornell-AI-finance-lectures.git
cd eCornell-AI-finance-lectures

# 2. Open a session directory and launch Julia
cd lectures/session-1
julia

# 3. In the Julia REPL, activate and instantiate the environment
using Pkg
Pkg.activate(".")
Pkg.instantiate()

# 4. Open the lecture notebook in VS Code or Jupyter and run cells sequentially
```

Each session has its own `Project.toml` that depends on the shared [`eCornellAIFinance`](code/) package. The first `Pkg.instantiate()` will download all dependencies (including `JumpHMM.jl` from GitHub).

## Sessions

### [Session 1: When Optimal Fails — Stress-Testing Classical Minimum-Variance Portfolios](lectures/session-1/)
Build a classical Markowitz portfolio from synthetic data, then systematically break it. Derive portfolio reward and risk from log growth rates, solve the quadratic program, compute the efficient frontier, and run AI-powered stress tests (correlation shifts, price drops, trading costs) with Monte Carlo confidence bands. Produces the baseline scorecard that benchmarks all later improvements.

| Notebook | Description |
|:---------|:------------|
| [Lecture](lectures/session-1/eCornell-AI-Finance-S1-Lecture-StressTestingMinVariancePortfolios-May-2026.ipynb) | Theory: growth rates, reward/risk, estimation, Markowitz QP, efficient frontier, three weaknesses, stress testing framework |
| [Example 1](lectures/session-1/eCornell-AI-Finance-S1-Example-BuildMinVariancePortfolio-May-2026.ipynb) | Build the portfolio: synthetic data, estimation, efficient frontier, input sensitivity |
| [Example 2](lectures/session-1/eCornell-AI-Finance-S1-Example-StressTestMinVariancePortfolio-May-2026.ipynb) | Stress test: correlation shifts, price drops, Monte Carlo simulation, baseline scorecard |

### [Session 2: From Static Weights to Adaptive Allocation — Designing the AI Rebalancing Engine](lectures/session-2/)
Replace variance minimization with utility maximization. Build a Cobb-Douglas allocator driven by EMA crossover sentiment and Single Index Model fundamentals. Compare to CES and log-linear alternatives. Wire the allocator into a daily rebalancing engine with drawdown limits, turnover caps, and de-risking protocols.

| Notebook | Description |
|:---------|:------------|
| [Lecture](lectures/session-2/eCornell-AI-Finance-S2-Lecture-AIRebalancingEngine-May-2026.ipynb) | Theory: utility maximization, Cobb-Douglas/CES/log-linear, EMA sentiment, SIM preferences, trigger rules |
| [Example 1](lectures/session-2/eCornell-AI-Finance-S2-Example-BuildCobbDouglasAllocator-May-2026.ipynb) | Build the allocator: synthetic market data, preference weights, utility comparison |
| [Example 2](lectures/session-2/eCornell-AI-Finance-S2-Example-RebalancingEngineScorecard-May-2026.ipynb) | Run the engine: rebalancing loop, scorecard vs. benchmarks, sensitivity analysis |

### [Session 3: Can It Survive Regime Shifts? HMM Backtesting and Bandit Challengers](lectures/session-3/)
Stress-test the rebalancing engine against realistic regime-switching data. Fit a JumpHMM model that captures volatility clustering, fat tails, and regime persistence. Backtest across hundreds of Monte Carlo paths. Introduce an epsilon-greedy combinatorial bandit as an adaptive challenger. Produce a formal validation report with pass/fail criteria.

| Notebook | Description |
|:---------|:------------|
| [Lecture](lectures/session-3/eCornell-AI-Finance-S3-Lecture-HMMBacktesting-May-2026.ipynb) | Theory: JumpHMM, regime switching, Monte Carlo backtesting, combinatorial bandits, validation criteria |
| [Example 1](lectures/session-3/eCornell-AI-Finance-S3-Example-HMMBacktesting-May-2026.ipynb) | Fit HMM, generate scenarios, backtest the Cobb-Douglas engine across 100 paths |
| [Example 2](lectures/session-3/eCornell-AI-Finance-S3-Example-BanditChallengerValidation-May-2026.ipynb) | Bandit challenger, head-to-head comparison, formal validation report |

### [Session 4: From Prototype to Production — Sentiment, Triggers, and Portfolio Operations](lectures/session-4/)
Combine the bandit and Cobb-Douglas allocator into a production system. Add synthetic sentiment monitoring with lambda overrides, three escalation triggers (drawdown, sentiment crash, bandit churn), and human override protocols. Build an operational dashboard and stress-test with crash injection.

| Notebook | Description |
|:---------|:------------|
| [Lecture](lectures/session-4/eCornell-AI-Finance-S4-Lecture-ProductionOps-May-2026.ipynb) | Theory: production loop, sentiment monitoring, escalation triggers, investment committee transparency |
| [Example 1](lectures/session-4/eCornell-AI-Finance-S4-Example-ProductionPortfolioEngine-May-2026.ipynb) | Run the production loop: bandit + Cobb-Douglas, sentiment override, escalation log |
| [Example 2](lectures/session-4/eCornell-AI-Finance-S4-Example-OperationalDashboard-May-2026.ipynb) | Operational dashboard, multi-path simulation, crash stress test |
